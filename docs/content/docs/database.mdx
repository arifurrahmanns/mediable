Every `mediable()` instance talks to **one** database. Four built-in providers plus an escape hatch for everything else.

```ts
database: {
  provider: 'postgres',                                    // 'sqlite' | 'postgres' | 'mysql' | 'mongodb'
  connection: { url: process.env.DATABASE_URL! },
  autoMigrate: true,
}
```

## Provider matrix

| Provider | `connection` shape | Peer module loaded |
|---|---|---|
| `sqlite` | `{ filename: './storage/media.db' }` *or* `{ url: 'file:./storage/media.db' }` | `better-sqlite3` |
| `postgres` | `{ url: 'postgresql://user:pass@host:5432/db' }` | `pg` |
| `mysql` | `{ url: 'mysql://user:pass@host:3306/db' }` | `mysql2` |
| `mongodb` | `{ url: 'mongodb://host:27017/db' }` | `mongoose` |

All four peers ship as regular dependencies of `mediable` — `pnpm add mediable` installs them. You don't need separate `pnpm add pg` commands.

## `autoMigrate`

```ts
database: { provider: 'postgres', connection: { url: '...' }, autoMigrate: true }
```

With `autoMigrate: true` the `media` table is created on the first request if it doesn't exist — using `CREATE TABLE IF NOT EXISTS`. Works for SQLite, Postgres, MySQL. For MongoDB, indexes are created on first connect.

**For production**, consider running the migration upfront instead of racing on first request:

```bash
npx mediable migrate
```

That loads your `media.ts` config and applies the schema explicitly. You can still keep `autoMigrate: true` as a safety net.

## Schema

The built-in `media` table:

```sql
CREATE TABLE media (
  id                    TEXT PRIMARY KEY,
  uuid                  TEXT UNIQUE NOT NULL,
  model_type            TEXT NOT NULL,
  model_id              TEXT NOT NULL,
  collection_name       TEXT NOT NULL,
  name                  TEXT NOT NULL,
  file_name             TEXT NOT NULL,
  mime_type             TEXT NOT NULL,
  disk                  TEXT NOT NULL,
  conversions_disk      TEXT NOT NULL,
  size                  BIGINT NOT NULL,
  manipulations         JSONB NOT NULL,
  custom_properties     JSONB NOT NULL,
  generated_conversions JSONB NOT NULL,
  responsive_images     JSONB NOT NULL,
  order_column          INTEGER NOT NULL DEFAULT 0,
  status                TEXT NOT NULL DEFAULT 'ready',
  optimized_at          TIMESTAMPTZ,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX media_owner_idx ON media (model_type, model_id);
CREATE INDEX media_owner_collection_idx ON media (model_type, model_id, collection_name);
CREATE INDEX media_status_idx ON media (status, created_at);
```

The JSON columns store structured data Postgres/MySQL/SQLite can query if you need to:

- `manipulations` — conversion plan metadata
- `custom_properties` — freeform metadata you set via `media.addMedia({ customProperties: { alt: '…' } })`
- `generated_conversions` — `{ "thumb": true, "preview": false }` flags per conversion
- `responsive_images` — srcset data for responsive-image plugin

## Direct-upload lifecycle columns

Two columns specifically for the [direct-to-storage flow](/docs/direct-upload):

- `status` — `'pending'` between presign and confirm, `'ready'` when the upload is committed, `'failed'` for unrecoverable errors.
- `optimized_at` — timestamp set when every configured conversion has finished.

Pending records are invisible to `media.list()` / `media.url()` — reap with:

```ts
const abandoned = await repo.findOrphans(new Date(Date.now() - 3600_000))
for (const record of abandoned) {
  await media.delete(record.id)
}
```

Run this as a cron every 15 minutes.

## Bring your own adapter

If you want mediable to share your app's Prisma / Drizzle / Kysely connection, implement `DatabaseAdapter`:

```ts
import type { DatabaseAdapter } from 'mediable'

const myAdapter: DatabaseAdapter = {
  id: 'my-orm',
  async create(model, data) { /* ... */ },
  async findOne(model, where) { /* ... */ },
  async findMany(model, query) { /* ... */ },
  async update(model, where, data) { /* ... */ },
  async delete(model, where) { /* ... */ },
}

export const media = mediable({
  database: myAdapter,
  // ...
})
```

The built-in adapters are thin — see the source for examples.

## Switching providers

Going from SQLite in dev to Postgres in prod is a one-block change:

```ts
// dev
database: {
  provider: 'sqlite',
  connection: { filename: './storage/media.db' },
  autoMigrate: true,
}

// prod
database: {
  provider: 'postgres',
  connection: { url: process.env.DATABASE_URL! },
  autoMigrate: true,
}
```

Existing data doesn't migrate automatically — if you need to move data, dump + import between SQLite and Postgres with standard tools (or write a script against `media.list()` and `media.attach()`).
